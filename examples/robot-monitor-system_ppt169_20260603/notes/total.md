# 01_cover

大家好，今天我要介绍的是机器人监控系统。这是一个基于三层架构的机器人实时监控与故障排查平台，旨在解决工业机器人运行状态无法实时掌握、故障排查困难的问题。整个系统采用硬件安全兜底、大脑本地决策、平台辅助决策的三级分层决策架构，能够实现对机器人运行状态的全面监控和智能处置。

---

# 02_architecture

接下来介绍系统的总体架构。整个系统采用经典的三层架构设计，具备数据采集和指令控制的双向能力。上层是应用层，即Web监控平台，负责平台辅助决策。提供PC网页端和移动端H5两种访问方式，包含实时监控、异常告警、故障溯源、远程控制和辅助决策分析五大核心功能。对于隐性劣化、复杂关联、疑难故障等复杂异常，平台通过分析三十天以上的历史数据趋势，调用故障知识库，给出诊断建议。中间是服务层，采用Java后端加MySQL数据库，实时存储机器人的运行数据、任务记录和故障日志，同时负责指令下发管理和知识库分析。底层是设备采集层，包含机器人控制柜和机器人大脑。控制柜对接不同品牌机器人，采集启停状态、关节电流温度等关键数据。机器人大脑作为本地实时决策核心，能处理百分之九十以上的运行异常，实现毫秒级响应。特别重要的是硬件安全回路，这是最高优先级的安全防线，急停按钮、安全光栅、碰撞传感器、过流短路等触发时，纯硬件电路直接切断动力电源，不经过大脑软件，确保人身和设备安全。

# 03_features

现在介绍平台的关键功能。第一个是实时状态查看功能，系统提供机器人总览看板、单机器人详情页和任务运行日志，每两秒轮询刷新，实时同步机器人的现场运行状态，并自动记录每次任务的开始时间、结束时间和运行结果。页面清晰展示空闲、执行任务、暂停和异常停机四种状态。第二个是异常告警模块，当机器人触发故障时，后端识别异常后会在Web页面弹窗红色告警，同时通过短信和APP消息推送通知相关人员，告警信息包含故障代码、发生时间和对应的任务编号。第三个是故障排查溯源功能，这是本系统的创新亮点。系统内置故障码数据库，可自动匹配故障原因；支持调取故障前五分钟的实时参数曲线，对比正常工况快速定位问题；还能按月统计故障频次，汇总高频故障辅助预防性维护。所有异常记录永久入库归档，工程师无需去现场即可定位故障诱因。

# 04_control_flow

最后介绍三级分层决策架构的详细设计。第一级是硬件安全回路决策，具有最高优先级。当急停按钮被按下、安全光栅或安全门被触发、碰撞传感器硬触发、主电路过流或短路时，硬件继电器直接断开动力电源，机器人瞬间停机。这个过程完全不经过大脑软件逻辑，大脑和平台只能接收急停触发的状态信号，无法干预或取消。这是工业机器人安全设计的强制性要求，避免因软件故障导致安全保护失效。第二级是机器人大脑本地决策，处理百分之九十以上的运行异常。大脑将异常分为三级：一级轻微异常如网络丢包、温度漂移，本地容错补偿后继续运行，仅上报日志；二级中度异常如负载超标、轨迹偏移，暂停当前任务等待平台指令，同时上报异常代码；三级严重异常如超温、编码器中断，紧急停机并锁存故障数据。大脑决策的优势在于实时性、数据本地化、网络中断时仍可保护。第三级是平台辅助决策，处理复杂隐性异常。包括轴承磨损等隐性劣化异常、多机器人协同冲突等复杂关联异常、无明确代码的疑难故障、以及需要与MES或ERP联动的跨系统协调异常。平台与大脑的协同流程是：大脑上报异常原始数据，平台调用历史库和知识库分析诊断，运维人员确认决策后下发处置指令，大脑执行并反馈结果。这种三级架构既保证了致命异常的毫秒级响应，又解决了复杂异常需要人工经验判断的问题。

# 05_navigation_baseline

接下来汇报导航Agent的进展。目前我们的导航Agent已经完成基础版实现，采用LLM加静态场景图的技术路线。简单说，机器人现在能跑通了。核心链路是：用户说我喝果汁，LLM解析意图，从静态场景图中匹配目标位置，调用底层导航接口让机器人移动到目标坐标，到达后任务结束。但这个基础版存在七大局限。第一，场景图是静态封闭词汇，用户说找充电宝时LLM匹配不到，只能随机选择，导致无法识别未定义物体。第二，规划是扁平的，LLM直接从全量viewpoint中选容易产生幻觉，准确率只有约百分之七十。第三，无动态重规划，走到一半遇到障碍物只能从头规划，响应慢效率低。第四，到达后无视觉核验，机器人到了就说完成但不确认目标是否在那里，导致到了但没找到的投诉多。第五，未知环境盲目探索，目标不在场景图中时机器人原地转圈，探索效率极低。第六，LLM缺乏常识约束，用户说渴了可能导航到厕所而非吧台，用户体验差。第七，Token成本高，每步都调LLM，一百米走廊要调十次，成本高且响应慢每步三到五秒。结论是基础版能跑通核心闭环，但存在准确率、可靠性、成本三大瓶颈，需要系统性优化。

# 06_optimization_roadmap

我们调研了二零二二到二零二六年导航领域的十三篇里程碑论文，制定了四阶段优化路径。第一阶段是工程守底，预计一周，目标是让机器人跑得稳。引入SayCan和Grounded Decoding的安全约束思想，包括结构化输出校验、可执行性评分、关键字段兜底。解决幻觉选错和格式崩坏问题，预期将失败率从百分之十到十五降低到百分之三以内。第二阶段是架构升级，预计两周，目标是让机器人跑得准。引入SayPlan、ConceptGraphs和VLMaps的经典架构，包括层级规划、开放词汇场景图、空间语言理解。解决封闭词汇和扁平规划问题，预期将准确率从百分之七十提升到百分之九十。第三阶段是闭环感知，预计两周，目标是让机器人跑得实。引入SayNav、CoW和PixNav的闭环感知机制，包括动态重规划、到达后视觉核验、像素引导精调。解决无重规划和无核验问题，预期用户投诉率降低百分之五十。第四阶段是降本增效，预计两周，目标是让机器人跑得快跑得省。引入HiRO-Nav、VLFM和ESC的效率优化机制，包括混合推理、定向探索、常识推理注入。解决盲目探索、缺常识和成本高问题，预期Token成本降低百分之六十到八十，响应速度提升五倍。整体路径是从能跑通到跑得稳、跑得准、跑得实、跑得快，预计总工期七周。以上是汇报内容，请各位领导指导。

# 07_code_review

最后汇报代码审查发现。我们对Robot Monitor System进行了全面的代码质量审计，共发现十三项问题，覆盖严重、中等、建议三个等级，同时项目已有五十三个单元测试用例。严重问题共四项：第一，数据类定义问题，status.py中history字段类型注解不完整，应该使用List[Dict[str, str]]而非裸list；第二，错误处理不完善，nodes.py中JSON解析缺少异常处理，LLM输出格式错误会导致程序直接崩溃；第三，硬编码配置问题，LLM服务URL和场景图路径硬编码在代码中，应抽取到配置文件或环境变量；第四，并发安全问题，running_tasks字典缺少锁保护，多协程并发访问可能产生竞态条件。中等问题共五项：资源泄漏风险，异常路径下status_map可能未清理；重复代码，person_plan_node和obj_plan_node存在大量重复逻辑；未使用的代码和注释；Prompt管理问题，存在未使用的Prompt和拼写错误；监控节点逻辑问题，轮询间隔硬编码。建议改进共四项：路由映射不一致、测试代码混入主文件、日志记录使用print而非logging、类型注解不完整。这些问题已记录，后续迭代中将逐步修复。
